## 🤖 Instructions for AI (Antigravity / Cursor / Windsurf)

As an AI coding assistant in this repository, you **MUST** follow these rules:

1. **Check the Index First**: Before starting any task, refer to [00-MASTER-INDEX.md](file:///c:/AI%20Starter/docs/00-MASTER-INDEX.md) to understand the active features, database schemas, mockups, and codebase mapping.
2. **Context Alignment**:
   - When asked to modify or build a feature `F[XX]`, read the corresponding specification file at `docs/features/F[XX]-[nama-fitur].md`.
   - Cross-reference the database tables and columns using [01-master-schema.sql](file:///c:/AI%20Starter/docs/database/01-master-schema.sql). Do not guess table columns or SQL data types.
   - Cross-reference the layout design using the HTML mockup in `docs/mockups/F[XX]-[nama-fitur].html`.
3. **Change Tracking & Git Diffs**:
   - When a specification file under `docs/features/` changes, analyze the Git diff to find out exactly what Business Criteria or Technical Specification changed.
   - Proactively recommend updates to `src/` and `tests/` based on those changes.
4. **Writing Tests**:
   - Ensure every feature implementation has corresponding automated test cases written under `tests/` that cover all the Acceptance Criteria and QA Scenarios listed in the feature's Markdown file.
5. **No Code Without Spec**:
   - Do not implement logic that deviates from the specification. If there is a contradiction between the code requested and the specs, raise a question to the user immediately.
6. **SQL & Database Design Standards**:
   - **No Physical Foreign Keys**: Do NOT generate physical `FOREIGN KEY` constraints in DDL/SQL files. Table relationships must be managed logically in application models, not database constraints.
   - **Unique Index on Every Table**: Every table must have a unique index (single or composite) to prevent redundant data and act as a reliable logical key.
   - **Parent Unique Key in Child Tables**: Child tables must explicitly include the unique key of their parent table (e.g., store `user_email` or `user_username` along with `user_id` in child tables) to reduce lookup overhead and guarantee logical integrity.
